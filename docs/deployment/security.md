# WBSB Security Controls

## Overview

This document describes the security controls implemented in Iteration 11 (Security Hardening & Production Readiness). These controls protect the feedback webhook (`POST /feedback`) — the only inbound HTTP surface in WBSB.

Before I11, the feedback endpoint accepted any request and validated only the payload body. After I11, every request is cryptographically authenticated, rate-limited, and replay-protected.

---

## Threat Model

| Threat | Control | Task |
|---|---|---|
| Unauthenticated feedback submission | HMAC-SHA256 request signing | I11-1, I11-5 |
| Replay attack (captured request resent) | Nonce deduplication (TTL=10min) + timestamp freshness window (±300s) | I11-2, I11-5 |
| Brute-force / flooding | Per-IP rate limit (10/60s + burst 3) + global circuit breaker (100/60s) | I11-3, I11-5 |
| Credential leak via log files | Pseudonymized IPs; no secret/signature/body in log output | I11-4, I11-5 |
| Container privilege escalation | Non-root user UID 1000 in Docker image | I11-6 |
| Feedback file read by other OS processes | File permissions 0o600 on artifacts; 0o700 on feedback directory | I11-6 |
| Error responses leaking internals | Sanitized error responses — no stack traces, file paths, or module names | I11-5 |
| Vulnerable dependencies | pip-audit CI gate (fails on HIGH/CRITICAL CVEs) | I11-7 |
| Build tool exposure in production image | Multi-stage Docker build (builder stage isolated from production) | I11-7 |
| TLS bypass via HTTP downgrade | X-Forwarded-Proto enforcement (`WBSB_REQUIRE_HTTPS=true`) | I11-5 |

## Security Controls Detail

### HMAC-SHA256 Authentication (I11-1, I11-5)

Every `POST /feedback` request must include three headers:

| Header | Value |
|---|---|
| `X-WBSB-Timestamp` | Unix timestamp (integer string) |
| `X-WBSB-Signature` | HMAC-SHA256 hex digest (lowercase) |
| `X-WBSB-Nonce` | UUID4 string |

**Signing string format:**
```
{timestamp}.{request_body_utf8}
```

**Signature computation:**
```python
import hmac, hashlib
signing = f"{timestamp}.{body.decode('utf-8')}"
sig = hmac.new(secret.encode(), signing.encode(), hashlib.sha256).hexdigest()
```

The comparison uses `hmac.compare_digest` — constant-time, prevents timing attacks.

### Timestamp Freshness (I11-1, I11-5)

Requests are rejected if `abs(now - timestamp) > 300 seconds`. The ±5 minute window covers legitimate clock skew while preventing captured-request replay.

### Nonce Replay Prevention (I11-2, I11-5)

Each nonce (UUID4) is recorded in an in-memory store for 10 minutes. A repeated nonce within that window returns HTTP 409. Store capacity: 10,000 entries with LRU eviction. Does not survive process restart — timestamp freshness is the fallback during the first 5 minutes after restart.

### Rate Limiting (I11-3, I11-5)

- **Per-IP:** 10 requests per 60-second sliding window + burst allowance of 3 (effective: 13 before HTTP 429)
- **Global:** 100 requests per 60-second window → HTTP 503
- Both limits are fail-open: a bug in the limiter allows the request through rather than blocking legitimate traffic

### Request Handling Order

```
0. X-Forwarded-Proto check    → HTTP 400 if http and WBSB_REQUIRE_HTTPS=true
1. Rate limit check           → HTTP 429 (per-IP) or HTTP 503 (global)
2. Parse auth headers         → HTTP 401 if any absent or nonce malformed
3. Timestamp freshness        → HTTP 401 if expired or non-integer
4. HMAC verification          → HTTP 401 if invalid
5. Nonce replay check         → HTTP 409 if already seen within TTL
6. Body validation            → HTTP 400 (existing I9 logic)
7. Feedback storage           → HTTP 200
```

### Runtime Hardening (I11-6)

- Docker container runs as UID 1000 (`wbsb` user, non-root)
- Feedback artifact files: `chmod 0o600` (owner read-write only)
- Feedback directory: `chmod 0o700` (owner access only)

### Supply Chain Security (I11-7)

- Dependencies pinned via `requirements.lock` (generated by `pip-compile`, not `pip freeze`)
- `pip-audit` CI job fails the build on any HIGH or CRITICAL CVE
- Multi-stage Docker build: build tools never reach the production image
- `trivy` image scan in CI on every push

### Security Observability (I11-4, I11-5)

Every security event (auth failure, replay, rate limit, feedback received, invalid input) emits a structured log entry. IP addresses are pseudonymized before logging (last octet zeroed for IPv4, last group zeroed for IPv6). The following values are **never** logged: `secret`, `signature`, `request body`, `comment`, full nonce value.

## Environment Variables

| Variable | Required | Description |
|---|---|---|
| `WBSB_FEEDBACK_SECRET` | Yes (production) | Shared secret for HMAC signing. Must be set before starting the server in production mode. |
| `WBSB_ENV` | No (default: `production`) | Set to `development` to bypass HMAC auth. Never use in production. |
| `WBSB_REQUIRE_HTTPS` | No (default: unset) | Set to `true` to reject requests where `X-Forwarded-Proto: http`. Enable when behind Caddy/nginx TLS termination. |

**Starting the server:**
```bash
# Production (HMAC enforced)
WBSB_FEEDBACK_SECRET=<secret> WBSB_REQUIRE_HTTPS=true wbsb feedback serve

# Development (HMAC bypassed — local testing only)
WBSB_ENV=development wbsb feedback serve
```

## Out of Scope (I11)

The following are explicitly not addressed in I11:

- OAuth 2.0 / OpenID Connect / SSO
- Multi-tenant RBAC
- Automatic secret rotation (Key Vault integration)
- SIEM integration or log forwarding
- Formal penetration testing
- WAF configuration
- DDoS mitigation beyond in-process rate limiting
- Web dashboard security (I8 scope)
