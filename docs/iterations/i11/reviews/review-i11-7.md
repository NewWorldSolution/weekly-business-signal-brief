## Project Context

WBSB is a deterministic analytics engine. I11-7 adds dependency pinning via `pip-compile`, `pip-audit` and `trivy` CI jobs, multi-stage Docker build, and `cap_drop: ALL` in docker-compose.

**Non-root user preservation:** I11-6 added `USER wbsb` (UID 1000) to the Dockerfile. I11-7 converts it to multi-stage. The non-root user must survive this conversion — it must appear in the production stage, not just the builder stage.

**`pyproject.toml` constraint:** Abstract version specifiers (`>=`, `~=`) must not be changed to exact pins (`==`). Only `pip-tools` should be added to dev dependencies. Any other specifier change → `CHANGES REQUIRED`.

## Review Execution Steps

### Step 1 — Checkout

```bash
git fetch origin
git checkout feature/i11-7-supply-chain
git pull origin feature/i11-7-supply-chain
```

### Step 2 — Run validation

```bash
pytest --tb=short -q
ruff check .
```

If either fails: `CHANGES REQUIRED`.

### Step 3 — Scope check

```bash
git diff --name-only feature/iteration-11
```

Allowed: `requirements.lock`, `Dockerfile`, `docker-compose.yml`, `.github/workflows/security.yml`, `pyproject.toml`.

Forbidden: `src/wbsb/feedback/store.py`, `src/wbsb/feedback/server.py`, `src/wbsb/cli.py`, `config/rules.yaml`, any test file.

### Step 4 — Lockfile tool check

```bash
head -5 requirements.lock
```

Must contain `pip-compile` header comment (e.g. `# This file is autogenerated by pip-compile with Python 3.11`). If it looks like `pip freeze` output (contains `-e .` or local paths) → `CHANGES REQUIRED`.

```bash
grep -n "\-e \.\|file://" requirements.lock
```

Must return nothing. Editable installs do not belong in a lock file.

### Step 5 — pip-audit check (local)

```bash
pip install pip-audit
pip-audit --requirement requirements.lock --fail-on HIGH
```

Must pass with zero HIGH or CRITICAL CVEs.

### Step 6 — `pyproject.toml` specifier check

```bash
git diff feature/iteration-11 -- pyproject.toml
```

Only acceptable changes:
- Adding `pip-tools` to dev/optional dependencies
- No other lines changed

```bash
grep -n "pip-tools" pyproject.toml
```

Must find `pip-tools` in a dev/optional deps section.

```bash
grep -n "==[0-9]" pyproject.toml
```

No direct dependency pinned with `==` in pyproject.toml (lockfile handles pinning).

### Step 7 — Multi-stage Dockerfile check

```bash
grep -n "^FROM\|AS builder\|AS production" Dockerfile
```

Must find two `FROM` lines — `AS builder` and `AS production` (or equivalent names).

### Step 8 — Non-root user in production stage

```bash
grep -n "USER wbsb\|useradd.*wbsb\|chown.*wbsb" Dockerfile
```

`USER wbsb` and `useradd` must appear in the production stage (after the second `FROM`), not only in the builder stage.

### Step 9 — No build tools in production stage

```bash
grep -n "gcc\|build-essential\|apt-get install" Dockerfile
```

If any build toolchain installs appear in the production stage → `MAJOR`. Build tools in the builder stage are acceptable (they are not copied to production).

### Step 10 — `cap_drop` in docker-compose.yml

```bash
grep -n "cap_drop\|- ALL" docker-compose.yml
```

Must find `cap_drop: [ALL]` or the equivalent YAML list form for each service.

### Step 11 — CI workflow check

```bash
cat .github/workflows/security.yml
```

Verify:
- Both `pip-audit` and `trivy` jobs present
- `pip-audit` uses `--fail-on HIGH`
- `trivy` uses `severity: CRITICAL` and `exit-code: 1`
- Triggers on `push` and `pull_request`

```bash
python -c "import yaml; yaml.safe_load(open('.github/workflows/security.yml'))"
```

Must parse without error.
